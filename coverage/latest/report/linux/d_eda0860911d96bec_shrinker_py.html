37def sort_key(buffer): 

38 """Returns a sort key such that "simpler" buffers are smaller than 

39 "more complicated" ones. 

40 

41 We define sort_key so that x is simpler than y if x is shorter than y or if 

42 they have the same length and x < y lexicographically. This is called the 

43 shortlex order. 

44 

45 The reason for using the shortlex order is: 

46 

47 1. If x is shorter than y then that means we had to make fewer decisions 

48 in constructing the test case when we ran x than we did when we ran y. 

49 2. If x is the same length as y then replacing a byte with a lower byte 

50 corresponds to reducing the value of an integer we drew with draw_bits 

51 towards zero. 

52 3. We want a total order, and given (2) the natural choices for things of 

53 the same size are either the lexicographic or colexicographic orders 

54 (the latter being the lexicographic order of the reverse of the string). 

55 Because values drawn early in generation potentially get used in more 

56 places they potentially have a more significant impact on the final 

57 result, so it makes sense to prioritise reducing earlier values over 

58 later ones. This makes the lexicographic order the more natural choice. 

59 """ 

60 return (len(buffer), buffer)

109class Shrinker: 

110 """A shrinker is a child object of a ConjectureRunner which is designed to 

111 manage the associated state of a particular shrink problem. That is, we 

112 have some initial ConjectureData object and some property of interest 

113 that it satisfies, and we want to find a ConjectureData object with a 

114 shortlex (see sort_key above) smaller buffer that exhibits the same 

115 property. 

116 

117 Currently the only property of interest we use is that the status is 

118 INTERESTING and the interesting_origin takes on some fixed value, but we 

119 may potentially be interested in other use cases later. 

120 However we assume that data with a status < VALID never satisfies the predicate. 

121 

122 The shrinker keeps track of a value shrink_target which represents the 

123 current best known ConjectureData object satisfying the predicate. 

124 It refines this value by repeatedly running *shrink passes*, which are 

125 methods that perform a series of transformations to the current shrink_target 

126 and evaluate the underlying test function to find new ConjectureData 

127 objects. If any of these satisfy the predicate, the shrink_target 

128 is updated automatically. Shrinking runs until no shrink pass can 

129 improve the shrink_target, at which point it stops. It may also be 

130 terminated if the underlying engine throws RunIsComplete, but that 

131 is handled by the calling code rather than the Shrinker. 

132 

133 ======================= 

134 Designing Shrink Passes 

135 ======================= 

136 

137 Generally a shrink pass is just any function that calls 

138 cached_test_function and/or incorporate_new_buffer a number of times, 

139 but there are a couple of useful things to bear in mind. 

140 

141 A shrink pass *makes progress* if running it changes self.shrink_target 

142 (i.e. it tries a shortlex smaller ConjectureData object satisfying 

143 the predicate). The desired end state of shrinking is to find a 

144 value such that no shrink pass can make progress, i.e. that we 

145 are at a local minimum for each shrink pass. 

146 

147 In aid of this goal, the main invariant that a shrink pass much 

148 satisfy is that whether it makes progress must be deterministic. 

149 It is fine (encouraged even) for the specific progress it makes 

150 to be non-deterministic, but if you run a shrink pass, it makes 

151 no progress, and then you immediately run it again, it should 

152 never succeed on the second time. This allows us to stop as soon 

153 as we have run each shrink pass and seen no progress on any of 

154 them. 

155 

156 This means that e.g. it's fine to try each of N deletions 

157 or replacements in a random order, but it's not OK to try N random 

158 deletions (unless you have already shrunk at least once, though we 

159 don't currently take advantage of this loophole). 

160 

161 Shrink passes need to be written so as to be robust against 

162 change in the underlying shrink target. It is generally safe 

163 to assume that the shrink target does not change prior to the 

164 point of first modification - e.g. if you change no bytes at 

165 index ``i``, all examples whose start is ``<= i`` still exist, 

166 as do all blocks, and the data object is still of length 

167 ``>= i + 1``. This can only be violated by bad user code which 

168 relies on an external source of non-determinism. 

169 

170 When the underlying shrink_target changes, shrink 

171 passes should not run substantially more test_function calls 

172 on success than they do on failure. Say, no more than a constant 

173 factor more. In particular shrink passes should not iterate to a 

174 fixed point. 

175 

176 This means that shrink passes are often written with loops that 

177 are carefully designed to do the right thing in the case that no 

178 shrinks occurred and try to adapt to any changes to do a reasonable 

179 job. e.g. say we wanted to write a shrink pass that tried deleting 

180 each individual byte (this isn't an especially good choice, 

181 but it leads to a simple illustrative example), we might do it 

182 by iterating over the buffer like so: 

183 

184 .. code-block:: python 

185 

186 i = 0 

187 while i < len(self.shrink_target.buffer): 

188 if not self.incorporate_new_buffer( 

189 self.shrink_target.buffer[:i] + self.shrink_target.buffer[i + 1 :] 

190 ): 

191 i += 1 

192 

193 The reason for writing the loop this way is that i is always a 

194 valid index into the current buffer, even if the current buffer 

195 changes as a result of our actions. When the buffer changes, 

196 we leave the index where it is rather than restarting from the 

197 beginning, and carry on. This means that the number of steps we 

198 run in this case is always bounded above by the number of steps 

199 we would run if nothing works. 

200 

201 Another thing to bear in mind about shrink pass design is that 

202 they should prioritise *progress*. If you have N operations that 

203 you need to run, you should try to order them in such a way as 

204 to avoid stalling, where you have long periods of test function 

205 invocations where no shrinks happen. This is bad because whenever 

206 we shrink we reduce the amount of work the shrinker has to do 

207 in future, and often speed up the test function, so we ideally 

208 wanted those shrinks to happen much earlier in the process. 

209 

210 Sometimes stalls are inevitable of course - e.g. if the pass 

211 makes no progress, then the entire thing is just one long stall, 

212 but it's helpful to design it so that stalls are less likely 

213 in typical behaviour. 

214 

215 The two easiest ways to do this are: 

216 

217 * Just run the N steps in random order. As long as a 

218 reasonably large proportion of the operations succeed, this 

219 guarantees the expected stall length is quite short. The 

220 book keeping for making sure this does the right thing when 

221 it succeeds can be quite annoying. 

222 * When you have any sort of nested loop, loop in such a way 

223 that both loop variables change each time. This prevents 

224 stalls which occur when one particular value for the outer 

225 loop is impossible to make progress on, rendering the entire 

226 inner loop into a stall. 

227 

228 However, although progress is good, too much progress can be 

229 a bad sign! If you're *only* seeing successful reductions, 

230 that's probably a sign that you are making changes that are 

231 too timid. Two useful things to offset this: 

232 

233 * It's worth writing shrink passes which are *adaptive*, in 

234 the sense that when operations seem to be working really 

235 well we try to bundle multiple of them together. This can 

236 often be used to turn what would be O(m) successful calls 

237 into O(log(m)). 

238 * It's often worth trying one or two special minimal values 

239 before trying anything more fine grained (e.g. replacing 

240 the whole thing with zero). 

241 

242 """

366 def incorporate_new_buffer(self, buffer): 

367 """Either runs the test function on this buffer and returns True if 

368 that changed the shrink_target, or determines that doing so would 

369 be useless and returns False without running it.""" 

370 

371 buffer = bytes(buffer[: self.shrink_target.index]) 

372 # Sometimes an attempt at lexicographic minimization will do the wrong 

373 # thing because the buffer has changed under it (e.g. something has 

374 # turned into a write, the bit size has changed). The result would be 

375 # an invalid string, but it's better for us to just ignore it here as 

376 # it turns out to involve quite a lot of tricky book-keeping to get 

377 # this right and it's better to just handle it in one place. 

378 if sort_key(buffer) >= sort_key(self.shrink_target.buffer): 

379 return False 

380 

381 if self.shrink_target.buffer.startswith(buffer): 

382 return False 

383 

384 previous = self.shrink_target 

385 self.cached_test_function(buffer) 

386 return previous is not self.shrink_target

420 def shrink(self): 

421 """Run the full set of shrinks and update shrink_target. 

422 

423 This method is "mostly idempotent" - calling it twice is unlikely to 

424 have any effect, though it has a non-zero probability of doing so. 

425 """ 

426 # We assume that if an all-zero block of bytes is an interesting 

427 # example then we're not going to do better than that. 

428 # This might not technically be true: e.g. for integers() | booleans() 

429 # the simplest example is actually [1, 0]. Missing this case is fairly 

430 # harmless and this allows us to make various simplifying assumptions 

431 # about the structure of the data (principally that we're never 

432 # operating on a block of all zero bytes so can use non-zeroness as a 

433 # signpost of complexity). 

434 if not any(self.shrink_target.buffer) or self.incorporate_new_buffer( 

435 bytes(len(self.shrink_target.buffer)) 

436 ): 

437 return 

438 

439 try: 

440 self.greedy_shrink() 

441 except StopShrinking: 

442 pass 

443 finally: 

444 if self.engine.report_debug_info: 

445

447 return "s" if n != 1 else "" 

448 

449 total_deleted = self.initial_size - len(self.shrink_target.buffer) 

450 

451 self.debug("---------------------") 

452 self.debug("Shrink pass profiling") 

453 self.debug("---------------------") 

454 self.debug("") 

455 calls = self.engine.call_count - self.initial_calls 

456 self.debug( 

457 "Shrinking made a total of %d call%s " 

458 "of which %d shrank. This deleted %d byte%s out of %d."

528 def fixate_shrink_passes(self, passes): 

529 """Run steps from each pass in ``passes`` until the current shrink target 

530 is a fixed point of all of them.""" 

531 passes = list(map(self.shrink_pass, passes)) 

532 

533 any_ran = True 

534 while any_ran: 

535 any_ran = False 

536 

537 reordering = {} 

538 

539 # We run remove_discarded after every pass to do cleanup 

540 # keeping track of whether that actually works. Either there is 

541 # no discarded data and it is basically free, or it reliably works 

542 # and deletes data, or it doesn't work. In that latter case we turn 

543 # it off for the rest of this loop through the passes, but will 

544 # try again once all of the passes have been run. 

545 can_discard = self.remove_discarded() 

546 

547 calls_at_loop_start = self.calls 

548 

549 # We keep track of how many calls can be made by a single step 

550 # without making progress and use this to test how much to pad 

551 # out self.max_stall by as we go along. 

552 max_calls_per_failing_step = 1 

553 

554 for sp in passes: 

555 if can_discard: 

556 can_discard = self.remove_discarded() 

557 

558 before_sp = self.shrink_target 

559 

560 # Run the shrink pass until it fails to make any progress 

561 # max_failures times in a row. This implicitly boosts shrink 

562 # passes that are more likely to work. 

563 failures = 0 

564 max_failures = 20 

565 while failures < max_failures: 

566 # We don't allow more than max_stall consecutive failures 

567 # to shrink, but this means that if we're unlucky and the 

568 # shrink passes are in a bad order where only the ones at 

569 # the end are useful, if we're not careful this heuristic 

570 # might stop us before we've tried everything. In order to 

571 # avoid that happening, we make sure that there's always 

572 # plenty of breathing room to make it through a single 

573 # iteration of the fixate_shrink_passes loop. 

574 self.max_stall = max( 

575 self.max_stall, 

576 2 * max_calls_per_failing_step 

577 + (self.calls - calls_at_loop_start), 

578 ) 

579 

580 prev = self.shrink_target 

581 initial_calls = self.calls 

582 # It's better for us to run shrink passes in a deterministic 

583 # order, to avoid repeat work, but this can cause us to create 

584 # long stalls when there are a lot of steps which fail to do 

585 # anything useful. In order to avoid this, once we've noticed 

586 # we're in a stall (i.e. half of max_failures calls have failed 

587 # to do anything) we switch to randomly jumping around. If we 

588 # find a success then we'll resume deterministic order from 

589 # there which, with any luck, is in a new good region. 

590 if not sp.step(random_order=failures >= max_failures // 2): 

591 # step returns False when there is nothing to do because 

592 # the entire choice tree is exhausted. If this happens 

593 # we break because we literally can't run this pass any 

594 # more than we already have until something else makes 

595 # progress. 

596 break 

597 any_ran = True 

598 

599 # Don't count steps that didn't actually try to do 

600 # anything as failures. Otherwise, this call is a failure 

601 # if it failed to make any changes to the shrink target. 

602 if initial_calls != self.calls: 

603 if prev is not self.shrink_target: 

604 failures = 0 

605 else: 

606 max_calls_per_failing_step = max( 

607 max_calls_per_failing_step, self.calls - initial_calls 

608 ) 

609 failures += 1 

610 

611 # We reorder the shrink passes so that on our next run through 

612 # we try good ones first. The rule is that shrink passes that 

613 # did nothing useful are the worst, shrink passes that reduced 

614 # the length are the best. 

615 if self.shrink_target is before_sp: 

616 reordering[sp] = 1 

617 elif len(self.buffer) < len(before_sp.buffer): 

618 reordering[sp] = -1 

619 else: 

620 reordering[sp] = 0 

621 

622 passes.sort(key=reordering.__getitem__)

653 @defines_shrink_pass() 

654 def pass_to_descendant(self, chooser): 

655 """Attempt to replace each example with a descendant example. 

656 

657 This is designed to deal with strategies that call themselves 

658 recursively. For example, suppose we had: 

659 

660 binary_tree = st.deferred( 

661 lambda: st.one_of( 

662 st.integers(), st.tuples(binary_tree, binary_tree))) 

663 

664 This pass guarantees that we can replace any binary tree with one of 

665 its subtrees - each of those will create an interval that the parent 

666 could validly be replaced with, and this pass will try doing that. 

667 

668 This is pretty expensive - it takes O(len(intervals)^2) - so we run it 

669 late in the process when we've got the number of intervals as far down 

670 as possible. 

671 """ 

672 

673 label = chooser.choose( 

674 self.distinct_labels, lambda l: len(self.examples_by_label[l]) >= 2 

675 ) 

676 

677 ls = self.examples_by_label[label] 

678 

679 i = chooser.choose(range(len(ls) - 1)) 

680 

681 ancestor = ls[i] 

682 

683 if i + 1 == len(ls) or ls[i + 1].start >= ancestor.end: 

684 return 

685 

686 @self.cached(label, i) 

687 def descendants(): 

688 lo = i + 1 

689 hi = len(ls) 

690 while lo + 1 < hi: 

691 mid = (lo + hi) // 2 

692 if ls[mid].start >= ancestor.end: 

693 hi = mid 

694 else: 

695 lo = mid 

696 return [t for t in ls[i + 1 : hi] if t.length < ancestor.length] 

697 

698 descendant = chooser.choose(descendants, lambda ex: ex.length > 0) 

699 

700 assert ancestor.start <= descendant.start 

701 assert ancestor.end >= descendant.end 

702 assert descendant.length < ancestor.length 

703 

704 self.incorporate_new_buffer( 

705 self.buffer[: ancestor.start] 

706 + self.buffer[descendant.start : descendant.end] 

707 + self.buffer[ancestor.end :] 

708 )

710 def lower_common_block_offset(self): 

711 """Sometimes we find ourselves in a situation where changes to one part 

712 of the byte stream unlock changes to other parts. Sometimes this is 

713 good, but sometimes this can cause us to exhibit exponential slow 

714 downs! 

715 

716 e.g. suppose we had the following: 

717 

718 m = draw(integers(min_value=0)) 

719 n = draw(integers(min_value=0)) 

720 assert abs(m - n) > 1 

721 

722 If this fails then we'll end up with a loop where on each iteration we 

723 reduce each of m and n by 2 - m can't go lower because of n, then n 

724 can't go lower because of m. 

725 

726 This will take us O(m) iterations to complete, which is exponential in 

727 the data size, as we gradually zig zag our way towards zero. 

728 

729 This can only happen if we're failing to reduce the size of the byte 

730 stream: The number of iterations that reduce the length of the byte 

731 stream is bounded by that length. 

732 

733 So what we do is this: We keep track of which blocks are changing, and 

734 then if there's some non-zero common offset to them we try and minimize 

735 them all at once by lowering that offset. 

736 

737 This may not work, and it definitely won't get us out of all possible 

738 exponential slow downs (an example of where it doesn't is where the 

739 shape of the blocks changes as a result of this bouncing behaviour), 

740 but it fails fast when it doesn't work and gets us out of a really 

741 nastily slow case when it does. 

742 """ 

743 if len(self.__changed_blocks) <= 1: 

744 return 

745 

746 current = self.shrink_target 

747 

748 blocked = [current.buffer[u:v] for u, v in self.all_block_bounds()] 

749 

750 changed = [ 

751 i 

752 for i in sorted(self.__changed_blocks) 

753 if not self.shrink_target.blocks[i].trivial 

754 ] 

755 

756 if not changed: 

757 return 

758 

759 ints = [int_from_bytes(blocked[i]) for i in changed] 

760 offset = min(ints) 

761 assert offset > 0 

762 

763 for i in range(len(ints)): 

764 ints[i] -= offset 

765 

766 def reoffset(o): 

767 new_blocks = list(blocked) 

768 for i, v in zip(changed, ints): 

769 new_blocks[i] = int_to_bytes(v + o, len(blocked[i])) 

770 return self.incorporate_new_buffer(b"".join(new_blocks))

782 @property 

783 def __changed_blocks(self): 

784 if self.__last_checked_changed_at is not self.shrink_target: 

785 prev_target = self.__last_checked_changed_at 

786 new_target = self.shrink_target 

787 assert prev_target is not new_target 

788 prev = prev_target.buffer 

789 new = new_target.buffer 

790 assert sort_key(new) < sort_key(prev) 

791 

792 if ( 

793 len(new_target.blocks) != len(prev_target.blocks) 

794 or new_target.blocks.endpoints != prev_target.blocks.endpoints 

795 ): 

796 self.__all_changed_blocks = set() 

797 else: 

798 blocks = new_target.blocks

801 # by checking if the tail past this point has been modified.

805 lambda i: prev[blocks.start(i) :] != new[blocks.start(i) :],

808 # Index of the first block whose contents have been changed, 

809 # because we know that this predicate is true for zero (because 

810 # the prefix from the start is empty), so the result must be True 

811 # for the bytes from the start of this block and False for the 

812 # bytes from the end, hence the change is in this block. 

813 first_changed = binary_search( 

814 0, 

815 len(blocks), 

816 lambda i: prev[: blocks.start(i)] == new[: blocks.start(i)],

820 # check if any specific block values have changed.

822 u, v = blocks.bounds(i) 

823 if i not in self.__all_changed_blocks and prev[u:v] != new[u:v]: 

824 self.__all_changed_blocks.add(i)

850 def try_shrinking_blocks(self, blocks, b): 

851 """Attempts to replace each block in the blocks list with b. Returns 

852 True if it succeeded (which may include some additional modifications 

853 to shrink_target). 

854 

855 In current usage it is expected that each of the blocks currently have 

856 the same value, although this is not essential. Note that b must be 

857 < the block at min(blocks) or this is not a valid shrink. 

858 

859 This method will attempt to do some small amount of work to delete data 

860 that occurs after the end of the blocks. This is useful for cases where 

861 there is some size dependency on the value of a block. 

862 """ 

863 initial_attempt = bytearray(self.shrink_target.buffer) 

864 for i, block in enumerate(blocks): 

865 if block >= len(self.blocks): 

866 blocks = blocks[:i] 

867 break 

868 u, v = self.blocks[block].bounds 

869 n = min(self.blocks[block].length, len(b)) 

870 initial_attempt[v - n : v] = b[-n:] 

871 

872 if not blocks: 

873 return False 

874 

875 start = self.shrink_target.blocks[blocks[0]].start 

876 end = self.shrink_target.blocks[blocks[-1]].end 

877 

878 initial_data = self.cached_test_function(initial_attempt) 

879 

880 if initial_data is self.shrink_target: 

881 self.lower_common_block_offset() 

882 return True 

883 

884 # If this produced something completely invalid we ditch it 

885 # here rather than trying to persevere. 

886 if initial_data.status < Status.VALID: 

887 return False 

888 

889 # We've shrunk inside our group of blocks, so we have no way to 

890 # continue. (This only happens when shrinking more than one block at 

891 # a time). 

892 if len(initial_data.buffer) < v: 

893 return False 

894 

895 lost_data = len(self.shrink_target.buffer) - len(initial_data.buffer) 

896 

897 # If this did not in fact cause the data size to shrink we 

898 # bail here because it's not worth trying to delete stuff from 

899 # the remainder. 

900 if lost_data <= 0: 

901 return False 

902 

903 # We now look for contiguous regions to delete that might help fix up 

904 # this failed shrink. We only look for contiguous regions of the right 

905 # lengths because doing anything more than that starts to get very 

906 # expensive. See minimize_individual_blocks for where we 

907 # try to be more aggressive. 

908 regions_to_delete = {(end, end + lost_data)} 

909 

910 for j in (blocks[-1] + 1, blocks[-1] + 2): 

911 if j >= min(len(initial_data.blocks), len(self.blocks)): 

912 continue 

913 # We look for a block very shortly after the last one that has 

914 # lost some of its size, and try to delete from the beginning so 

915 # that it retains the same integer value. This is a bit of a hyper 

916 # specific trick designed to make our integers() strategy shrink 

917 # well. 

918 r1, s1 = self.shrink_target.blocks[j].bounds 

919 r2, s2 = initial_data.blocks[j].bounds 

920 lost = (s1 - r1) - (s2 - r2) 

921 # Apparently a coverage bug? An assert False in the body of this 

922 # will reliably fail, but it shows up as uncovered. 

923 if lost <= 0 or r1 != r2: # pragma: no cover 

924 continue 

925 regions_to_delete.add((r1, r1 + lost)) 

926 

927 for ex in self.shrink_target.examples: 

928 if ex.start > start: 

929 continue 

930 if ex.end <= end: 

931 continue 

932 

933 replacement = initial_data.examples[ex.index] 

934 

935 in_original = [c for c in ex.children if c.start >= end] 

936 

937 in_replaced = [c for c in replacement.children if c.start >= end] 

938 

939 if len(in_replaced) >= len(in_original) or not in_replaced: 

940 continue 

941 

942 # We've found an example where some of the children went missing 

943 # as a result of this change, and just replacing it with the data 

944 # it would have had and removing the spillover didn't work. This 

945 # means that some of its children towards the right must be 

946 # important, so we try to arrange it so that it retains its 

947 # rightmost children instead of its leftmost. 

948 regions_to_delete.add( 

949 (in_original[0].start, in_original[-len(in_replaced)].start) 

950 ) 

951 

952 for u, v in sorted(regions_to_delete, key=lambda x: x[1] - x[0], reverse=True): 

953 try_with_deleted = bytearray(initial_attempt) 

954 del try_with_deleted[u:v] 

955 if self.incorporate_new_buffer(try_with_deleted): 

956 return True 

957 return False

959 def remove_discarded(self): 

960 """Try removing all bytes marked as discarded. 

961 

962 This is primarily to deal with data that has been ignored while 

963 doing rejection sampling - e.g. as a result of an integer range, or a 

964 filtered strategy. 

965 

966 Such data will also be handled by the adaptive_example_deletion pass, 

967 but that pass is necessarily more conservative and will try deleting 

968 each interval individually. The common case is that all data drawn and 

969 rejected can just be thrown away immediately in one block, so this pass 

970 will be much faster than trying each one individually when it works. 

971 

972 returns False if there is discarded data and removing it does not work, 

973 otherwise returns True. 

974 """ 

975 while self.shrink_target.has_discards: 

976 discarded = [] 

977 

978 for ex in self.shrink_target.examples: 

979 if ( 

980 ex.length > 0 

981 and ex.discarded 

982 and (not discarded or ex.start >= discarded[-1][-1]) 

983 ): 

984 discarded.append((ex.start, ex.end)) 

985 

986 # This can happen if we have discards but they are all of 

987 # zero length. This shouldn't happen very often so it's 

988 # faster to check for it here than at the point of example 

989 # generation. 

990 if not discarded: 

991 break 

992 

993 attempt = bytearray(self.shrink_target.buffer) 

994 for u, v in reversed(discarded): 

995 del attempt[u:v] 

996 

997 if not self.incorporate_new_buffer(attempt): 

998 return False 

999 return True

1285 def run_block_program(self, i, description, original, repeats=1): 

1286 """Block programs are a mini-DSL for block rewriting, defined as a sequence 

1287 of commands that can be run at some index into the blocks 

1288 

1289 Commands are: 

1290 

1291 * "-", subtract one from this block. 

1292 * "X", delete this block 

1293 

1294 If a command does not apply (currently only because it's - on a zero 

1295 block) the block will be silently skipped over. 

1296 

1297 This method runs the block program in ``description`` at block index 

1298 ``i`` on the ConjectureData ``original``. If ``repeats > 1`` then it 

1299 will attempt to approximate the results of running it that many times. 

1300 

1301 Returns True if this successfully changes the underlying shrink target, 

1302 else False. 

1303 """ 

1304 if i + len(description) > len(original.blocks) or i < 0: 

1305 return False 

1306 attempt = bytearray(original.buffer) 

1307 for _ in range(repeats): 

1308 for k, d in reversed(list(enumerate(description))): 

1309 j = i + k 

1310 u, v = original.blocks[j].bounds 

1311 if v > len(attempt): 

1312 return False 

1313 if d == "-": 

1314 value = int_from_bytes(attempt[u:v]) 

1315 if value == 0: 

1316 return False 

1317 else: 

1318 attempt[u:v] = int_to_bytes(value - 1, v - u) 

1319 elif d == "X": 

1320 del attempt[u:v] 

1321 else: 

1322 raise NotImplementedError(f"Unrecognised command {d!r}") 

1323 return self.incorporate_new_buffer(attempt)

1342@shrink_pass_family 

1343def block_program(self, chooser, description): 

1344 """Mini-DSL for block rewriting. A sequence of commands that will be run 

1345 over all contiguous sequences of blocks of the description length in order. 

1346 Commands are: 

1347 

1348 * ".", keep this block unchanged 

1349 * "-", subtract one from this block. 

1350 * "0", replace this block with zero 

1351 * "X", delete this block 

1352 

1353 If a command does not apply (currently only because it's - on a zero 

1354 block) the block will be silently skipped over. As a side effect of 

1355 running a block program its score will be updated. 

1356 """ 

1357 n = len(description) 

1358 

1359 """Adaptively attempt to run the block program at the current 

1360 index. If this successfully applies the block program ``k`` times 

1361 then this runs in ``O(log(k))`` test function calls.""" 

1362 i = chooser.choose(range(len(self.shrink_target.blocks) - n)) 

1363 # First, run the block program at the chosen index. If this fails, 

1364 # don't do any extra work, so that failure is as cheap as possible. 

1365 if not self.run_block_program(i, description, original=self.shrink_target): 

1366 return 

1367 

1368 # Because we run in a random order we will often find ourselves in the middle 

1369 # of a region where we could run the block program. We thus start by moving 

1370 # left to the beginning of that region if possible in order to to start from 

1371 # the beginning of that region. 

1372 def offset_left(k): 

1373 return i - k * n 

1374 

1375 i = offset_left( 

1376 find_integer( 

1377 lambda k: self.run_block_program( 

1378 offset_left(k), description, original=self.shrink_target 

1379 ) 

1380 ) 

1381 ) 

1382 

1383 original = self.shrink_target 

1384 

1385 # Now try to run the block program multiple times here. 

1386 find_integer( 

1387 lambda k: self.run_block_program(i, description, original=original, repeats=k) 

1388 )

1391@shrink_pass_family 

1392def dfa_replacement(self, chooser, dfa_name): 

1393 """Use one of our previously learned shrinking DFAs to reduce 

1394 the current test case. This works by finding a match of the DFA in the 

1395 current buffer that is not already minimal and attempting to replace it 

1396 with the minimal string matching that DFA. 

1397 """ 

1398 

1399 try: 

1400 dfa = SHRINKING_DFAS[dfa_name] 

1401 except KeyError: 

1402 dfa = self.extra_dfas[dfa_name] 

1403 

1404 matching_regions = self.matching_regions(dfa) 

1405 minimal = next(dfa.all_matching_strings()) 

1406 u, v = chooser.choose( 

1407 matching_regions, lambda t: self.buffer[t[0] : t[1]] != minimal 

1408 ) 

1409 p = self.buffer[u:v] 

1410 assert sort_key(minimal) < sort_key(p) 

1411 replaced = self.buffer[:u] + minimal + self.buffer[v:] 

1412 

1413 assert sort_key(replaced) < sort_key(self.buffer) 

1414 

1415 self.consider_new_buffer(replaced)

1430 def step(self, random_order=False): 

1431 tree = self.shrinker.shrink_pass_choice_trees[self] 

1432 if tree.exhausted: 

1433 return False 

1434 

1435 initial_shrinks = self.shrinker.shrinks 

1436 initial_calls = self.shrinker.calls 

1437 size = len(self.shrinker.shrink_target.buffer) 

1438 self.shrinker.engine.explain_next_call_as(self.name) 

1439 

1440 if random_order: 

1441 selection_order = random_selection_order(self.shrinker.random) 

1442 else: 

1443 selection_order = prefix_selection_order(self.last_prefix) 

1444 

1445 try: 

1446 self.last_prefix = tree.step( 

1447 selection_order, 

1448 lambda chooser: self.run_with_chooser(self.shrinker, chooser), 

1449 ) 

1450 finally: 

1451 self.calls += self.shrinker.calls - initial_calls 

1452 self.shrinks += self.shrinker.shrinks - initial_shrinks 

1453 self.deletions += size - len(self.shrinker.shrink_target.buffer) 

1454 self.shrinker.engine.clear_call_explanation() 

1455 return True
